# Developer's Codex of UX Mastery
## Accessibility as a Design Constraint

> **"Accessibility is not a checklist. It is a constraint that forces better, cleaner, and more robust UI design."**

## SECTION 1: Page Structure & Semantics (The Foundation)

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **1.1. Sequential Headings** | **Visual Hierarchy:** Just as you use font-weight and color to guide the eye, HTML tags establish the document's structural "skeleton." Skipping levels (e.g., H1 to H4) creates a disjointed outline, making the content feel "messy" to both machines and humans. Sequential headings create a clear information architecture that mirrors visual design principles. | ```html<br><h1 class="text-3xl font-bold">Page Title</h1><br><h2 class="text-2xl font-semibold">Section</h2><br><h3 class="text-xl font-medium">Subsection</h3>``` | ```html<br><h1>Title</h1><br><h4>Subtitle</h4>``` *(Skipped H2, H3)* |
| **1.2. Single H1 Per Page** | **Focal Point:** In design, you establish one primary focal point. The H1 is that focal point for content. Multiple H1s create visual confusion and dilute the page's purpose. A single H1 anchors the entire layout, just like a hero image or primary CTA. | ```html<br><h1 class="sr-only">Dashboard</h1><br><header><br>  <h2 class="text-2xl">Welcome Back</h2><br></header>``` | ```html<br><h1>Site Logo</h1><br><h1>Page Title</h1><br><h1>Banner</h1>``` |
| **1.3. Descriptive Headings** | **Cognitive Load:** Short, descriptive headings act as "signposts" that help users scan and understand content quickly. Long, vague headings force users to read entire paragraphs to understand context‚Äîincreasing cognitive load and reducing scanability. | ```html<br><h2>Order Confirmation</h2><br><h2>Shipping Address</h2>``` | ```html<br><h2>Information</h2><br><h2>Details About Your Purchase and What Happens Next</h2>``` |
| **1.4. Semantic Headings Only** | **Affordance:** Headings communicate structure, not decoration. Using `<p>` with classes for visual styling maintains semantic integrity while allowing design flexibility. This separation of concerns (structure vs. presentation) is fundamental to clean architecture. | ```html<br><h2>Section Title</h2><br><p class="text-2xl font-bold text-gray-400">Decorative Text</p>``` | ```html<br><h2>Section Title</h2><br><h2 class="text-gray-400">Decorative Text</h2>``` *(Misusing semantic tag)* |
| **1.5. Visual Heading Distinction** | **Vertical Rhythm:** Headings should create clear visual breaks in your typographic scale. At least 2 points larger than body text ensures headings establish proper vertical rhythm and guide the eye through content hierarchy. This is typography 101‚Äîaccessibility just enforces it. | ```css<br>:root {<br>  --text-base: 1rem;<br>  --text-h1: 2.5rem;<br>  --text-h2: 2rem;<br>  --text-h3: 1.5rem;<br>}<br>h1 { font-size: var(--text-h1); }<br>h2 { font-size: var(--text-h2); }<br>body { font-size: var(--text-base); }``` | ```css<br>h1 { font-size: 16px; }<br>body { font-size: 14px; }``` *(Insufficient distinction)* |
| **1.6. Semantic Landmarks** | **Information Architecture:** Landmarks (`<header>`, `<nav>`, `<main>`, `<aside>`, `<footer>`) create a mental map of the page. They're like chapters in a book‚Äîusers can jump directly to what they need. This reduces cognitive load and creates a more professional, organized experience. | ```html<br><header><br>  <nav aria-label="Main navigation">...</nav><br></header><br><main id="main-content">...</main><br><aside aria-label="Related links">...</aside><br><footer>...</footer>``` | ```html<br><div class="header">...</div><br><div class="content">...</div><br><div class="sidebar">...</div>``` *(No semantic structure)* |
| **1.7. Real Lists** | **Affordance & Visual Rhythm:** Real `<ul>` and `<ol>` elements provide native indentation, bullets/numbers, and list spacing. This creates consistent vertical rhythm. Using divs with custom styling breaks this affordance and requires more CSS to achieve what HTML provides naturally. | ```html<br><ul class="space-y-2"><br>  <li>First item</li><br>  <li>Second item</li><br></ul>``` | ```html<br><div class="list"><br>  <div class="item">‚Ä¢ First item</div><br>  <div class="item">‚Ä¢ Second item</div><br></div>``` |
| **1.8. Semantic Tables** | **Data Structure:** Tables are for tabular data, not layout. Using `<th>` and `<caption>` creates a clear data model that both humans and machines understand. This enforces proper information design‚Äîdata should be structured, not decorative. | ```html<br><table><br>  <caption>Monthly Sales Report</caption><br>  <thead><br>    <tr><th>Month</th><th>Revenue</th></tr><br>  </thead><br>  <tbody>...</tbody><br></table>``` | ```html<br><div class="table"><br>  <div class="row"><br>    <div class="cell">Month</div><br>  </div><br></div>``` *(Using divs for layout)* |
| **1.9. Unique IDs** | **Single Source of Truth:** Duplicate IDs break the fundamental web principle of unique identifiers. This creates unpredictable behavior and violates the DRY principle. Unique IDs ensure each element has a clear, unambiguous identity‚Äîessential for both styling and scripting. | ```html<br><label for="email-input">Email</label><br><input id="email-input" type="email" /><br><label for="phone-input">Phone</label><br><input id="phone-input" type="tel" />``` | ```html<br><label for="input">Email</label><br><input id="input" /><br><label for="input">Phone</label><br><input id="input" />``` *(Duplicate IDs)* |
| **1.10. Valid Code** | **Robustness:** Validation errors indicate broken relationships and unpredictable behavior. Clean, valid HTML is the foundation of reliable, maintainable code. It's like having a solid foundation before building a house‚Äîeverything else depends on it. | ```html<br><label for="email">Email</label><br><input id="email" type="email" aria-describedby="email-help" /><br><div id="email-help">Enter your email address</div>``` | ```html<br><label>Email</label><br><input aria-describedby="help" /><br><div id="help">Help text</div>``` *(Broken relationships)* |
| **1.11. Descriptive Page Title** | **Context & Branding:** The page title is the first thing users see in tabs, bookmarks, and search results. A unique, descriptive title provides immediate context and reinforces your brand. It's your first impression‚Äîmake it count. | ```html<br><title>Checkout - Your Cart | StoreName</title><br><title>Dashboard - Analytics Overview | AppName</title>``` | ```html<br><title>Page 1</title><br><title>Home</title>``` *(Generic, non-descriptive)* |
| **1.12. SPA Route Updates** | **State Communication:** In SPAs, route changes are invisible to assistive tech without explicit updates. Updating the title, moving focus, and announcing changes maintains the "mental model" of navigation. This creates a seamless, professional experience that feels native, not broken. | ```javascript<br>router.onRouteChange((route) => {<br>  document.title = `${route.title} | AppName`;<br>  const main = document.getElementById('main-content');<br>  main?.focus();<br>  main?.setAttribute('tabindex', '-1');<br>  announceToScreenReader(`Navigated to ${route.title}`);<br>});``` | ```javascript<br>router.onRouteChange((route) => {<br>  // No title update, no focus management<br>  renderComponent(route.component);<br>});``` |

## SECTION 2: Language and Text Handling

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **2.1. Language Declaration** | **Cultural Context:** Declaring language sets pronunciation, text direction, and cultural expectations. It's like setting the stage for a performance‚Äîeverything that follows makes more sense when the context is clear. | ```html<br><html lang="en"><br>  <head>...</head><br>  <body>...</body><br></html>``` | ```html<br><html><br>  <head>...</head><br>  <body>...</body><br></html>``` *(No lang attribute)* |
| **2.2. Inline Language Changes** | **Precision:** When content switches language mid-sentence, marking it ensures correct pronunciation and understanding. This attention to detail shows professionalism and respect for multilingual users. | ```html<br><p>Welcome to our <span lang="fr">caf√©</span>.</p><br><p>Read our <span lang="es">t√©rminos y condiciones</span>.</p>``` | ```html<br><p>Welcome to our caf√©.</p><br><p>Read our t√©rminos y condiciones.</p>``` *(No language marking)* |
| **2.3. Foreign Word Tagging** | **Pronunciation Accuracy:** Tagging foreign words ensures screen readers use the correct language synthesizer. This prevents jarring mispronunciations that break immersion and reduce comprehension. | ```html<br><p>The <span lang="de">Schadenfreude</span> was palpable.</p><br><p>She ordered <span lang="it">cappuccino</span>.</p>``` | ```html<br><p>The Schadenfreude was palpable.</p><br><p>She ordered cappuccino.</p>``` *(No language context)* |
| **2.4. Relative Text Units** | **Scalability & Vertical Rhythm:** Using `rem` and `em` creates a scalable typographic system. When users zoom, your entire design scales proportionally, maintaining visual harmony. `px` breaks this relationship and creates rigid, brittle layouts. | ```css<br>:root {<br>  --font-base: 1rem; /* 16px base */<br>  --font-sm: 0.875rem; /* 14px */<br>  --font-lg: 1.125rem; /* 18px */<br>}<br>body { font-size: var(--font-base); }<br>p { font-size: 1em; }<br>h1 { font-size: 2.5rem; }``` | ```css<br>body { font-size: 16px; }<br>p { font-size: 14px; }<br>h1 { font-size: 40px; }``` *(Fixed pixels)* |
| **2.5. Flexible Text Spacing** | **User Agency:** Allowing users to adjust spacing (line-height, letter-spacing, word-spacing) respects individual needs and preferences. Your design should be resilient, not rigid. This flexibility creates a more inclusive, professional experience. | ```css<br>p {<br>  line-height: 1.6;<br>  letter-spacing: 0.01em;<br>  word-spacing: 0.05em;<br>}<br>/* Respects user overrides */<br>@media (prefers-reduced-motion: no-preference) {<br>  * { transition: spacing 0.2s; }<br>}``` | ```css<br>p {<br>  line-height: 24px;<br>  letter-spacing: 0px;<br>  word-spacing: 0px;<br>}<br>/* Fixed, breaks with user overrides */``` |
| **2.6. Real Text Over Images** | **Maintainability & Performance:** Text in images can't be searched, translated, or styled. Real text with CSS gives you control, scalability, and better performance. It's the difference between a static poster and a dynamic, responsive design. | ```html<br><h1 class="text-4xl font-bold bg-gradient-to-r from-blue-600 to-purple-600 bg-clip-text text-transparent">Welcome</h1>``` | ```html<br><img src="welcome-text.png" alt="Welcome" />``` *(Text as image)* |

## SECTION 3: Forms, Inputs, and Errors

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **3.1. All Inputs Labeled** | **Affordance:** Labels make form fields discoverable and understandable. Without labels, users must guess what each field does‚Äîincreasing cognitive load and error rates. Labels are the "instructions" that make forms usable. | ```html<br><label for="email" class="block text-sm font-medium mb-1">Email Address</label><br><input id="email" type="email" class="w-full" />``` | ```html<br><input type="email" placeholder="Email" />``` *(No label)* |
| **3.2. Persistent Labels** | **Cognitive Ease:** Placeholders vanish when typing, forcing users to rely on short-term memory ("Was this field Email or Name?"). Persistent labels anchor the layout and reduce mental strain. They create visual stability in your form design. | ```html<br><label for="email" class="block text-sm font-medium mb-1">Email Address</label><br><input id="email" type="email" class="mt-1 block w-full" />``` | ```html<br><input type="email" placeholder="Email Address" />``` *(Placeholder only)* |
| **3.3. Label-Input Association** | **Programmatic Relationship:** The `for`/`id` relationship creates a clickable area that extends beyond the input itself. This increases the target size (Fitts's Law) and creates a better user experience. It's both accessible and more usable. | ```html<br><label for="email-input" class="block">Email</label><br><input id="email-input" type="email" />``` | ```html<br><label>Email</label><br><input type="email" />``` *(No association)* |
| **3.4. Unique Label IDs** | **Single Source of Truth:** Each form field needs a unique identifier. Duplicate IDs create ambiguity and break functionality. This is fundamental data integrity‚Äîeach piece of information needs a clear, unique address. | ```html<br><label for="email-primary">Primary Email</label><br><input id="email-primary" /><br><label for="email-secondary">Secondary Email</label><br><input id="email-secondary" />``` | ```html<br><label for="email">Email</label><br><input id="email" /><br><label for="email">Email</label><br><input id="email" />``` *(Duplicate IDs)* |
| **3.5. Fieldset Grouping** | **Visual & Cognitive Grouping:** `<fieldset>` and `<legend>` create both visual and semantic groups. This reduces cognitive load by chunking related information together. It's Gestalt's "proximity" principle applied to forms. | ```html<br><fieldset class="space-y-2"><br>  <legend class="text-lg font-semibold">Shipping Method</legend><br>  <label><input type="radio" name="shipping" /> Standard</label><br>  <label><input type="radio" name="shipping" /> Express</label><br></fieldset>``` | ```html<br><div><br>  <p>Shipping Method</p><br>  <label><input type="radio" name="shipping" /> Standard</label><br>  <label><input type="radio" name="shipping" /> Express</label><br></div>``` *(No semantic grouping)* |
| **3.6. Required Field Indication** | **Expectation Setting:** Users need to know what's mandatory before they start. Visual indicators (asterisks) and programmatic attributes (`required`) work together to set clear expectations. This reduces frustration and form abandonment. | ```html<br><label for="email" class="block">Email <span class="text-red-600" aria-label="required">*</span></label><br><input id="email" type="email" required aria-required="true" />``` | ```html<br><label>Email</label><br><input type="email" required />``` *(No visual indicator)* |
| **3.7. Help Text Association** | **Contextual Guidance:** Help text should be programmatically linked to its field so assistive tech can announce it. This creates a seamless, informative experience. It's like having a helpful assistant that provides context when needed. | ```html<br><label for="password">Password</label><br><input id="password" type="password" aria-describedby="password-help" /><br><div id="password-help" class="text-sm text-gray-600">Must be 8+ characters with numbers</div>``` | ```html<br><label>Password</label><br><input type="password" /><br><div>Must be 8+ characters</div>``` *(No association)* |
| **3.8. Autocomplete Attributes** | **Reducing Friction:** Specific `autocomplete` values help browsers fill forms accurately. This reduces typing, prevents errors, and creates a smoother experience. It's UX optimization through semantic HTML. | ```html<br><input type="text" autocomplete="email" /><br><input type="tel" autocomplete="tel" /><br><input type="text" autocomplete="address-line1" />``` | ```html<br><input type="text" /><br><input type="tel" /><br><input type="text" />``` *(No autocomplete hints)* |
| **3.9. Text-Based Error Messages** | **Information Design:** Color alone is insufficient because 8% of men are colorblind. Error messages must be text-based, clearly identifying the problem. This creates better information design that works for everyone. | ```html<br><div class="error-message" role="alert"><br>  <span class="icon" aria-hidden="true">‚ö†Ô∏è</span><br>  Email address is invalid<br></div>``` | ```html<br><input class="border-red-500" />``` *(Color only, no message)* |
| **3.10. Comprehensive Error Details** | **Problem-Solution Framework:** Good error messages identify the field, explain the problem, and suggest a fix. This follows the "problem-solution" pattern that reduces cognitive load and guides users to success. | ```html<br><div id="email-error" role="alert" class="text-red-600"><br>  Email field: Invalid format. Please enter an email like name@example.com<br></div>``` | ```html<br><div>Error</div>``` *(Vague, unhelpful)* |
| **3.11. Non-Intrusive Error Announcements** | **Respectful Communication:** Errors should be announced without stealing focus unexpectedly. Using `aria-invalid` and `role="alert"` allows assistive tech to announce errors without disrupting the user's flow. This creates a more professional, less jarring experience. | ```html<br><input type="email" aria-invalid="true" aria-describedby="email-error" /><br><div id="email-error" role="alert" class="sr-only">Email is invalid</div>``` | ```javascript<br>input.focus(); // Steals focus<br>alert("Error!"); // Blocks interaction``` |
| **3.12. Error Prevention Mechanisms** | **Trust & Safety:** For critical actions (financial, legal), provide review/confirmation steps. This builds trust and prevents costly mistakes. It's like having a safety net‚Äîusers feel more confident when they know they can review before committing. | ```html<br><section aria-labelledby="review-heading"><br>  <h2 id="review-heading">Review Your Order</h2><br>  <button type="button" onclick="editOrder()">Edit</button><br>  <button type="submit">Confirm Purchase</button><br></section>``` | ```html<br><button onclick="processPayment()">Pay Now</button>``` *(No review step)* |
| **3.13. Destructive Action Confirmation** | **Preventing Regret:** Delete actions are irreversible. Confirmation dialogs prevent accidental loss and give users a moment to reconsider. This is basic risk management in UX design. | ```javascript<br>function handleDelete() {<br>  if (confirm('Are you sure you want to delete this item? This cannot be undone.')) {<br>    deleteItem();<br>  }<br>}``` | ```javascript<br>function handleDelete() {<br>  deleteItem(); // No confirmation<br>}``` |

## SECTION 4: Hyperlinks and Links

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **4.1. Meaningful Link Text** | **Scanability:** Links should make sense out of context. Users often scan pages by tabbing through links. "Click here" provides no information‚Äîit's cognitive dead weight. Descriptive links create a better information architecture. | ```html<br><a href="/products">View our product catalog</a><br><a href="/contact">Contact our support team</a>``` | ```html<br><a href="/products">Click here</a><br><a href="/contact">Click here</a>``` *(Generic text)* |
| **4.2. Destination-Based Link Text** | **Predictability:** Using the destination page's H1 as link text creates consistency and predictability. Users know exactly where they're going. This reduces cognitive load and builds trust through transparency. | ```html<br><a href="/about">About Our Company</a><br><!-- Destination page has: <h1>About Our Company</h1> -->``` | ```html<br><a href="/about">Learn More</a><br><!-- Destination page has: <h1>About Our Company</h1> -->``` *(Mismatched text)* |
| **4.3. Visual Link Affordance** | **Discoverability:** Links must look like links. Underlines or strong contrast create visual affordance that guides users. Underlining non-links creates false affordances and confuses users. This is fundamental visual design. | ```css<br>a {<br>  color: var(--link-color);<br>  text-decoration: underline;<br>  text-underline-offset: 0.2em;<br>}<br>a:visited { color: var(--link-visited); }<br>a:hover { text-decoration-thickness: 2px; }``` | ```css<br>a { color: #333; text-decoration: none; }<br>.button { text-decoration: underline; }``` *(Links don't look like links)* |
| **4.4. File Type & Size Context** | **Expectation Management:** Users need to know what they're downloading. File type and size set expectations and help users prepare (e.g., "This is a large PDF"). This transparency builds trust and prevents frustration. | ```html<br><a href="/report.pdf" download>Annual Report <span class="text-sm">(PDF, 2.4 MB)</span></a><br><a href="/data.xlsx">Export Data <span class="text-sm">(Excel, 156 KB)</span></a>``` | ```html<br><a href="/report.pdf">Download Report</a>``` *(No context)* |
| **4.5. Unique Link Context** | **Information Architecture:** Duplicate link text going to different places creates ambiguity. If text must be repeated, add context (via `aria-label` or visible text) to distinguish destinations. This maintains clear information hierarchy. | ```html<br><a href="/product/1">Buy Now - Blue Widget</a><br><a href="/product/2">Buy Now - Red Widget</a><br><!-- OR --><br><a href="/product/1" aria-label="Buy Now for Blue Widget">Buy Now</a>``` | ```html<br><a href="/product/1">Buy Now</a><br><a href="/product/2">Buy Now</a>``` *(Ambiguous duplicates)* |
| **4.6. Context Change Warnings** | **User Agency:** Opening new windows or auto-submitting forms changes context without user consent. Warning users (via `target="_blank"` with `rel="noopener"` and visible indicators) respects user agency and prevents disorientation. | ```html<br><a href="/external" target="_blank" rel="noopener noreferrer"><br>  External Resource <span class="sr-only">(opens in new tab)</span><br>  <span aria-hidden="true">‚Üó</span><br></a>``` | ```html<br><a href="/external" target="_blank">External Resource</a>``` *(No warning, no rel)* |

## SECTION 5: Keyboard Navigation and Focus

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **5.1. Full Keyboard Operability** | **Universal Access:** Not everyone uses a mouse. Keyboard-only operation ensures your interface is accessible to all users and devices. This constraint forces you to design clear, logical interaction patterns. | ```javascript<br>button.addEventListener('keydown', (e) => {<br>  if (e.key === 'Enter' || e.key === ' ') {<br>    handleClick();<br>  }<br>});``` | ```javascript<br>button.addEventListener('click', handleClick);<br>// No keyboard support``` |
| **5.2. All Elements Reachable** | **Complete Functionality:** Every interactive element must be keyboard-accessible. If it's not reachable by Tab, it's not truly accessible. This ensures feature parity across input methods. | ```html<br><button>Click Me</button><br><a href="/link">Link</a><br><input type="text" /><br><!-- All naturally focusable -->``` | ```html<br><div onclick="doSomething()">Click Me</div><br><!-- Not keyboard accessible -->``` |
| **5.3. Logical Focus Order** | **Reading Flow:** Tab order should follow visual reading order (left-to-right, top-to-bottom). This creates a predictable, logical flow that matches users' visual scanning patterns. Breaking this creates cognitive dissonance. | ```html<br><nav><a href="/">Home</a></nav><br><main><br>  <h1>Title</h1><br>  <form>...</form><br></main>``` | ```html<br><main>...</main><br><nav>...</nav><br><!-- Visual order doesn't match tab order -->``` |
| **5.4. Skip to Main Content** | **Efficiency:** Skip links allow keyboard users to bypass repetitive navigation. This is like having a "fast-forward" button‚Äîit respects users' time and reduces frustration. It's a small detail that shows professional attention to UX. | ```html<br><a href="#main-content" class="skip-link">Skip to main content</a><br><nav>...</nav><br><main id="main-content" tabindex="-1">...</main>```<br>```css<br>.skip-link {<br>  position: absolute;<br>  top: -40px;<br>  left: 0;<br>}<br>.skip-link:focus { top: 0; }``` | ```html<br><nav>...</nav><br><main>...</main><br><!-- No skip link -->``` |
| **5.5. No Focus Traps** | **User Freedom:** Focus traps prevent users from navigating away, creating a frustrating, broken experience. Users must always be able to exit any component. This is fundamental to user agency. | ```javascript<br>modal.addEventListener('keydown', (e) => {<br>  if (e.key === 'Tab') {<br>    const focusable = modal.querySelectorAll('button, [href], input');<br>    const first = focusable[0];<br>    const last = focusable[focusable.length - 1];<br>    if (e.shiftKey && document.activeElement === first) {<br>      last.focus();<br>    } else if (!e.shiftKey && document.activeElement === last) {<br>      first.focus();<br>    }<br>  }<br>});``` | ```javascript<br>// Focus gets stuck in modal, can't escape``` |
| **5.6. Visible Focus Indicators** | **Feedback & Affordance:** Focus indicators show users where they are. High-contrast focus styles provide essential feedback. Removing them (`outline: none` without replacement) creates an invisible, confusing experience. | ```css<br>*:focus-visible {<br>  outline: 3px solid var(--focus-color);<br>  outline-offset: 2px;<br>  border-radius: 2px;<br>}<br>button:focus-visible {<br>  box-shadow: 0 0 0 3px var(--focus-color);<br>}``` | ```css<br>* { outline: none; }<br>/* No focus indicator */``` |
| **5.7. Focus Management** | **Context Preservation:** When opening modals/menus, focus should move logically into the component. When closing, it should return to the trigger. This maintains spatial context and prevents disorientation. | ```javascript<br>function openModal(trigger) {<br>  const modal = document.getElementById('modal');<br>  modal.showModal();<br>  const firstFocusable = modal.querySelector('button');<br>  firstFocusable?.focus();<br>  trigger.setAttribute('data-previous-focus', trigger.id);<br>}<br><br>function closeModal() {<br>  const trigger = document.querySelector('[data-previous-focus]');<br>  trigger?.focus();<br>  modal.close();<br>}``` | ```javascript<br>function openModal() {<br>  modal.show();<br>  // Focus stays on trigger, user is confused<br>}``` |
| **5.8. Arrow Key Navigation** | **Component Affordance:** Radio groups, tabs, and menus should use Arrow keys for internal navigation (not Tab). Tab moves between components; Arrows move within components. This creates intuitive, native-feeling interactions. | ```javascript<br>radioGroup.addEventListener('keydown', (e) => {<br>  const radios = Array.from(radioGroup.querySelectorAll('input'));<br>  const current = radios.indexOf(e.target);<br>  if (e.key === 'ArrowDown' || e.key === 'ArrowRight') {<br>    radios[(current + 1) % radios.length]?.focus();<br>  } else if (e.key === 'ArrowUp' || e.key === 'ArrowLeft') {<br>    radios[(current - 1 + radios.length) % radios.length]?.focus();<br>  }<br>});``` | ```javascript<br>// Using Tab inside radio group breaks expected behavior``` |

## SECTION 6: Color, Contrast, and Visual Cues

| The Tactic | The "Design Logic" (The Why) | Pro Code (Polished) | Amateur Code (Avoid) |
| :--- | :--- | :--- | :--- |
| **6.1. Text Contrast (4.5:1)** | **Readability:** Sufficient contrast ensures text is readable for everyone, including users with low vision or in bright sunlight. This isn't just accessibility‚Äîit's fundamental typography. | **6.2. Non-Text Contrast (3:1)** | **Discoverability:** Buttons, icons, and controls need sufficient contrast to be discoverable. Low contrast creates "ghost buttons" that are hard to find and use. This violates Fitts's Law and creates poor affordance. | ```css<br>button {<br>  background: var(--button-bg);<br>  color: var(--button-text);<br>  border: 2px solid var(--button-border);<br>  /* 3:1 minimum for borders/icons */<br>}<br>.icon {<br>  color: var(--icon-color); /* 3:1 minimum */<br>}``` | ```css<br>button {<br>  background: #f0f0f0;<br>  color: #e0e0e0; /* 1.2:1 - fails contrast */<br>}``` |
| **6.3. Accessible Brand Variants** | **Brand Flexibility:** If brand colors fail contrast, create accessible variants. This shows that you value both brand identity and user experience. It's a design constraint that forces creative solutions. | ```css<br>:root {<br>  --brand-primary: #0066cc;<br>  --brand-primary-dark: #004499; /* Accessible variant */<br>  --brand-primary-light: #3388dd; /* For dark backgrounds */<br>}<br>button.primary {<br>  background: var(--brand-primary-dark);<br>  color: white; /* 7.1:1 contrast */<br>}``` | ```css<br>button.primary {<br>  background: var(--brand-color); /* Fails contrast */<br>  color: white;<br>}``` |
| **6.4. Meaning Beyond Color** | **Information Redundancy:** Using color alone to convey meaning excludes 8% of men (colorblind). Adding icons, patterns, or text creates redundant information channels that work for everyone. This is better information design. | ```html<br><div class="status"><br>  <span class="icon" aria-hidden="true">‚ö†Ô∏è</span><br>  <span class="text">Warning: Low battery</span><br></div><br><!-- OR --><br><div class="chart"><br>  <div class="bar" style="height: 60%"><br>    <span class="label">Q1: 60%</span><br>  </div><br></div>``` | ```html<br><div class="status red">Warning</div><br><!-- Color only --><br><div class="chart"><br>  <div class="bar red"></div><br>  <div class="bar blue"></div><br></div><br><!-- No labels, color only -->``` |
| **6.5. Location-Independent Instructions** | **Universal Design:** Instructions shouldn't rely on layout ("click the blue button on the right"). Layouts change on mobile, and screen reader users don't have spatial context. Use descriptive, location-independent language. | ```html<br><button id="submit-btn" class="primary">Submit Form</button><br><!-- Instruction: "Click the Submit Form button" -->``` | ```html<br><button class="blue-button">Submit</button><br><!-- Instruction: "Click the blue button on the right" -->``` |
